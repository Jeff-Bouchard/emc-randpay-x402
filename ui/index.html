randpay is a probabilistic payment scheme for Emercoin, proposed for registration in the x402 foundation. At risk=0.01, 100 API calls produce one on-chain transaction — with mathematically identical expected revenue. No smart contracts. No gas token. No facilitator. emercoind is the facilitator.

Protocol flow

SERVER
01
randpay_mkchap amt risk t
Server calls emercoind. Returns CHAP hex — a single-use challenge bound to amount, risk, and timeout. Embedded in X-PAYMENT-REQUIRED header of the 402 response.
→
CLIENT
02
randpay_mktx "chap" t
Client runs this on their own emercoind node. Signs the CHAP and builds a probabilistic raw transaction. No funds move yet. Returns rawtx hex, sent back as X-PAYMENT-SIGNATURE.
→
SERVER
03
randpay_accept hexstring
Server calls emercoind with the rawtx. Probabilistic outcome is determined. Returns true (winner) or false (non-winner). Throws on invalid, expired, or replayed CHAP.
WINNER → 200 OK
randpay_accept returns true. emercoind broadcasts the winning tx. Server fulfills the request. X-PAYMENT-RESPONSE header confirms settlement. No additional step.
NON-WINNER → 402 (fresh CHAP)
randpay_accept returns false. No tx broadcast. No funds moved. Server returns 402 with a fresh CHAP embedded. Client retries immediately — no extra round-trip.

Risk rationale: 7-day uses risk=0.10 — short commitment, client retry cost is trivial, 10% miss rate acceptable. 30-day uses risk=0.03 — larger value at stake; at risk=0.10 a client could grind ~10 CHAPs before winning, effectively a discount attack at 30-day prices. At risk=0.03, expected non-winning attempts before success is ~1.03 — not exploitable.
